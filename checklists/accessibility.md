# Accessibility Checklist

Use this when a task changes UI, forms, navigation, modals, interactive controls, dashboards, or visual states.

## Acceptance Checks

- Interactive elements use semantic HTML where possible.
- Icon-only buttons have accessible labels.
- Inputs have labels and useful validation messages.
- Keyboard users can reach and operate the new UI.
- Focus states are visible.
- Modals, drawers, and menus handle focus reasonably.
- Colour is not the only way to understand state.
- Text has adequate contrast.
- Motion is restrained or respects reduced-motion expectations when relevant.
- Hit targets are large enough for touch when the UI is used on mobile.

## User Questions

- Can this be used with a keyboard?
- Do icon-only buttons have names?
- Do form fields have labels?
- What happens for users who cannot rely on colour alone?
- Does the UI still make sense on mobile or narrow screens?
