# AIDONE

**不要相信 "done"。要求证据。**

AIDONE 是一份面向产品验收的 AI 代码交付协议。它定义 AI coding agent 在声称任务完成之前，必须证明哪些事情。

适用场景：用户不逐行审代码，但仍然需要专业软件工程默认素养。

## 操作规则

AI 在声称代码任务完成前，必须报告：

1. 从产品角度看，改了什么。
2. 改了哪些文件。
3. 跑了哪些检查，以及结果。
4. 哪些 AIDONE 等级适用：P0、P1、相关 P2。
5. 还有哪些风险或未验证路径。
6. 用户如何手动验收。

没有新鲜证据时，不得说“完成了”“修好了”“可以了”“通过了”。证据可以来自测试、构建、类型检查、lint、浏览器检查、API 检查，或清楚描述过的手动验证。

## 等级

### P0 - 永远需要

P0 适用于所有非纯玩具代码变更，包括原型。

### P1 - 默认产品工程门槛

P1 适用于正常产品工作：功能、bugfix、UI、API、数据流、登录权限、支付、设置、管理后台，以及用户会持续使用的任何东西。

### P2 - 风险相关时启用

P2 适用于生产环境、数据迁移、外部服务、后台任务、支付、登录权限、用户数据、规模化或长期维护相关任务。

不要为了 P2 规则增加没有产品理由的复杂度。

## P0 - 安全和基本正确性

### 1. 代码里没有真实密钥

产品风险：API key、token、密码、数据库 URL、私钥、证书或服务凭证泄漏，会危害产品和用户。

AI 必须检查：

- 没有把真实密钥、token、密码、私钥或凭证写进代码、文档、示例、测试、截图或日志。
- 必需密钥来自环境变量、密钥管理器或项目已有配置系统。
- 新增配置时，`.env.example` 或等价文档只列变量名，不放真实值。

用户可以问：

- “这次新增了哪些环境变量或密钥？”
- “你有没有把真实 token 或凭证写到任何地方？”

### 2. 没有越权访问

产品风险：用户能看到、编辑、删除或导出不属于自己的数据。

AI 必须检查：

- 任何触碰用户数据的路由、API、页面、server action、数据库查询或管理操作，都有正确的登录和权限检查。
- 不能把“前端隐藏按钮”当成权限控制。
- 用户、租户、workspace、账号、组织、owner 等边界必须在服务端执行。

用户可以问：

- “谁能使用这个功能？”
- “权限在哪里检查？”
- “普通用户访问 admin 路径会发生什么？”

### 3. 用户输入经过校验

产品风险：坏输入会导致崩溃、脏数据、注入漏洞、坏 UI 或用户看不懂的错误。

AI 必须检查：

- 表单、query 参数、API body、上传文件、导入数据、webhook payload 在使用前被校验。
- 无效输入返回清楚、安全的错误。
- SQL、shell 命令、模板、文件路径、生成代码不能用不安全字符串拼接。

用户可以问：

- “你测试了哪些无效输入？”
- “校验失败时用户看到什么？”

### 4. 错误路径真实存在

产品风险：功能只在 happy path 能跑，真实使用中失败时静默、白屏或困惑。

AI 必须检查：

- 网络、API、数据库、文件、支付、登录、第三方服务失败时有处理。
- 用户看到安全、能理解的提示。
- 开发者能从日志、错误或 request id 排查。
- 不向用户暴露敏感内部细节。

用户可以问：

- “请求失败时发生什么？”
- “第三方服务挂了时发生什么？”
- “用户报错后，支持或工程应该看什么？”

### 5. 必须有验证证据

产品风险：AI 凭自信说完成，而不是凭证据说完成。

AI 必须检查：

- 运行相关检查：测试、lint、类型检查、构建、格式检查、API smoke test、浏览器验证，或项目里最接近的检查。
- 如果某个检查不能跑，说明原因，并提供替代验证。
- 不能用旧输出当新证据。

用户可以问：

- “你具体跑了哪些命令？”
- “什么通过了，什么失败了，什么没跑？”

## P1 - 默认产品工程门槛

### 6. 用户可见文案可维护

产品风险：以后改语言、语气、按钮、价格文案、法律文案、onboarding、通知会变慢且容易漏。

AI 必须检查：

- 如果项目已有 i18n、messages、copy、dictionary、locale 或内容系统，新用户可见文字不能散落在逻辑里。
- 按钮、标签、错误、空状态、toast、邮件、通知、页面标题、导航、表单校验都算用户可见文字。
- 如果项目没有文案系统，而这次新增了较多文案，应建立最小集中管理结构，或说明为什么这次不适合。

用户可以问：

- “列出这次新增的用户可见文案。”
- “这些文案放在哪里？”
- “以后换语言或改语气，是改一个文案区，还是要到组件里到处找？”

### 7. 有 loading / empty / error / success 状态

产品风险：数据慢、没数据、失败或部分可用时，UI 看起来像坏了。

AI 必须检查：

- 异步 UI 在相关场景下有 loading、empty、error、success 状态。
- 空状态说明发生了什么，用户下一步能做什么。
- 删除、清空、付款、发布等不可逆或高风险操作有确认、撤销或清楚后果说明。
- 长操作有进度或至少有进行中状态。

用户可以问：

- “怎么验证 loading、empty、error、success？”
- “等待时用户看到什么？”
- “没有数据时用户看到什么？”

### 8. 配置没有硬编码

产品风险：换环境、地区、模型、URL、价格、feature flag、限制或供应商，需要改代码重新发版。

AI 必须检查：

- API URL、服务端点、模型名、feature flag、地区、timeout、rate limit、价格常量、环境差异值没有随意硬编码。
- 必需配置缺失时应 fail fast，并给出清楚设置错误。
- 默认值安全且有文档。

用户可以问：

- “哪些值在本地、staging、production 会不同？”
- “这些值在哪里配置？”

### 9. 数据形状明确

产品风险：数据结构变化会在隐藏地方破坏产品。

AI 必须检查：

- 外部 API 响应、数据库记录、表单数据、webhook payload、核心业务对象有类型、schema、validator 或文档契约。
- 不用宽泛 `any`、未经检查的 cast 或松散对象绕过校验。
- 未知或无效数据被明确处理。

用户可以问：

- “这个功能依赖什么数据形状？”
- “API 少字段或字段异常时会怎样？”

### 10. 遵循项目已有模式

产品风险：代码库变成一堆不同风格，只有最后一个 agent 看得懂。

AI 必须检查：

- 路由、状态、表单、API 调用、样式、测试、日志、错误、文件结构沿用现有模式。
- 不新增库、框架、状态管理、UI kit、校验系统或架构模式，除非需求明确。
- 新依赖必须说明为什么现有技术栈不够。

用户可以问：

- “你是否遵循项目已有模式？”
- “有没有加新依赖或新约定？”
- “为什么必须加？”

### 11. 变更范围克制

产品风险：一个小需求变成大重写，带来用户没要求的回归。

AI 必须检查：

- diff 聚焦在用户要求的结果上。
- 不重构无关模块。
- 尽量分开行为变更和清理。
- 修改 legacy 或缺测试代码前，先描述当前行为，并保持小步变更。

用户可以问：

- “请求范围外改了什么？”
- “有没有包含无关重构？”

### 12. 测试覆盖行为

产品风险：测试通过，但产品行为仍然错。

AI 必须检查：

- 新功能、bugfix、校验、权限、重要边界和错误路径要加或更新测试。
- 测试优先描述用户可见或 API 可见行为。
- 修 bug 时尽量加回归测试。
- 如果无法测试，提供手动验收步骤，并说明缺口。

用户可以问：

- “哪些行为现在有测试覆盖？”
- “哪些重要路径只做了手动验证？”

### 13. 基础可访问性覆盖

产品风险：键盘、读屏、低视力或辅助工具用户无法使用产品。

AI 必须检查：

- 交互控件尽量使用语义元素。
- 只有图标的按钮有 accessible label。
- 输入框有 label 和有用的校验提示。
- focus、键盘操作、对比度、减少动画等基本预期被考虑。
- 颜色不是表达状态的唯一方式。

用户可以问：

- “这个功能能用键盘操作吗？”
- “图标按钮和表单字段有可访问名称吗？”

### 14. 必要文档已更新

产品风险：代码能跑，但没人知道怎么配置、使用、部署或恢复。

AI 必须检查：

- 行为、配置、命令、工作流变化时，更新 README、setup docs、env examples、API docs、runbooks 或 UI help。
- 不为显而易见的内部实现细节写文档。

用户可以问：

- “这个变更需要新的安装、部署或用户说明吗？”
- “文档在哪里？”

## P2 - 风险相关时启用

### 15. 兼容和迁移有计划

产品风险：已有用户、数据、链接、集成或保存设置在发布后坏掉。

AI 必须在相关时检查：

- 数据库迁移可回滚，或有回滚计划。
- 旧数据被迁移或保持兼容。
- 旧 URL、API 版本、设置、导入导出数据被处理。
- 高风险变更使用 feature flag 或分阶段发布。

用户可以问：

- “已有用户和已有数据会怎样？”
- “能回滚吗？”

### 16. 业务规则只有一个源头

产品风险：价格、权限、资格、额度、状态流转等规则复制到 UI、API、任务和测试里，之后彼此漂移。

AI 必须在相关时检查：

- 价格、权限、资格、quota、状态流转、校验、工作流规则有一个权威实现或契约。
- UI 可以为了体验镜像规则，但不能成为规则源头。

用户可以问：

- “这个规则的 source of truth 在哪里？”
- “同一规则有没有复制到别处？”

### 17. 后台任务可恢复

产品风险：job、queue、导入、导出、邮件、账单任务或 AI run 卡住或重复执行。

AI 必须在相关时检查：

- 后台任务尽量幂等。
- retry、timeout、取消、部分失败被处理。
- 用户或运维能看到长任务状态。

用户可以问：

- “这个任务中途失败会怎样？”
- “可以安全重试吗？”

### 18. 外部服务被隔离

产品风险：供应商改 API、宕机、限流或涨价时，影响扩散到整个产品。

AI 必须在相关时检查：

- 第三方 API 包在小 adapter 或 service boundary 后面。
- timeout、retry、rate limit、fallback、错误翻译被处理。
- 供应商对象不随意泄漏到全项目，除非项目明确接受这种耦合。

用户可以问：

- “以后换供应商，要改多少地方？”
- “供应商超时或限流时会怎样？”

## 交付模板

AI 完成代码任务后应使用：

```md
## Product Result

[从用户角度看改了什么。]

## Files Changed

- `path/to/file`: [为什么改]

## AIDONE Check

- P0 safety/correctness: [通过/缺口 + 细节]
- P1 product quality: [通过/缺口 + 细节]
- P2 risk areas: [不适用/通过/缺口]

## Verification

- `[command]`: [结果]
- Manual check: [步骤和结果]

## Remaining Risk

[没有，或列出具体未验证路径。]
```

填好的例子：

```md
## Product Result

Settings 现在可以把本地数据导出为 JSON。

## Files Changed

- `src/components/SettingsView.tsx`: 增加导出按钮和用户操作入口。
- `src/lib/db/export.ts`: 把本地记录整理成可导出的 JSON 结构。
- `src/lib/copy.ts`: 把新的 Settings 文案放进共享文案层。

## AIDONE Check

- P0 safety/correctness: 通过。没有新增密钥；导出只读取当前本地用户数据。
- P1 product quality: 部分通过。文案已集中管理，build 通过，但导出失败时的错误状态还缺。
- P2 risk areas: 不适用。当前是本地单机导出路径。

## Verification

- `pnpm build`: 通过。
- `pnpm lint`: 在 2 个无关旧文件上失败。
- Manual check: Settings -> Export Data，可以下载 JSON 文件。

## Remaining Risk

Clear Data 仍然是破坏性操作，在扩大用户测试前应该加二次确认。
```

## 用法

正常产品工作：

```text
Follow AIDONE.md. Use P0 and P1 by default. Apply P2 only where relevant.
```

快速原型：

```text
This is a prototype. Follow P0 only, and call out which P1 items are intentionally skipped.
```

高风险工作：

```text
Follow AIDONE.md with P0, P1, and relevant P2. Do not claim completion without verification evidence.
```
